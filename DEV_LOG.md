# 超市采购对账自动化系统 - 开发日志

## 项目概况

- **项目目标**：校内超市（约20000名学生）采购对账自动化，将出错率控制在千分之五以内
- **开发日期**：2026-03-17
- **数据批次**：3.16-对账汇总（10家供应商）
- **技术栈**：Python + 腾讯云OCR + openpyxl + PyMuPDF

---

## 系统架构

```
main.py                  ← 主程序入口，遍历所有供应商
├── ocr_parser.py        ← PDF送货单OCR解析模块
├── excel_reader.py      ← Excel文件读取模块（采购订单/入库单/进货单/退货单）
├── reconciler.py        ← 两阶段对账比对逻辑
└── report_generator.py  ← Excel对账报告生成
```

### 处理流程

```
每个供应商文件夹:
  PDF送货单 → OCR识别 → 结构化数据 ──┐
  采购入库单.xlsx → Excel读取 ────────┤→ 阶段一：采购对账（送货单 vs 入库单）
  进货单.xlsx → Excel读取 ────────────┤→ 阶段二：财务对账（进货单 vs 送货单 vs 退货单）
  退货单.xlsx → Excel读取 ────────────┘→ 生成对账报告Excel
```

## 各模块详细说明

### 1. OCR解析模块 (ocr_parser.py)

**腾讯云配置**：
- SecretId/SecretKey: 存储在 `SecretKey.csv`
- API: `RecognizeTableAccurateOCR` 统一使用（已去掉 GeneralAccurateOCR，API调用量减半）
- V3模式: `UseNewModel=true`，用于炬博和优亿家
- 太古: 可口可乐专用解析器（`_extract_taigu_items`）

**支持的送货单格式**（10家供应商有6种不同格式）：

| 供应商 | 表头格式 | 数量列 | 单价列 | 特殊处理 |
|--------|---------|--------|--------|---------|
| 承运 | 条码+品名合并/分开 | 送货数 | 单价 | √勾选符号 |
| 优品味 | 序号\|条码\|品名\|规格\|件数\|零数\|单价\|金额 | 零数 | 单价 | P0格式PO号 |
| 优亿家 | 商品名称\|条形码\|规格\|数量\|价格\|金额 | 数量 | 价格 | 无PO号 |
| 翔瑞 | 序号\|条形码\|商品名称\|商品单位\|订单数量\|销售价\|订单金额 | 订单数量 | 销售价 | 手写√标记 |
| 锐仕 | 序号\|商品条码\|商品名称\|规格\|件数\|散数\|件价\|散价\|金额 | 散数 | 散价 | 件价vs散价 |
| 炬博 | 商品条码\|商品名称\|规格\|整件数\|零散数\|单价\|金额\|单支总数量 | 单支总数量 | 单价(箱价) | 无表头页面 |
| 鑫凯 | 商品编码\|条形码\|商品名称\|规格\|单位\|数量\|单价\|金额 | 数量 | 单价 | 多表格长页 |
| 通力 | 类似承运格式 | 送货数 | 单价 | 跨年度数据 |
| 太古 | 可口可乐特殊格式，多表格嵌套 | 数量(N/M格式) | 单价(元) | 完全定制 |

**已实现的容错机制**：
- 条码13位校验，非标条码标记异常
- 数量为None时由金额÷单价反算
- 送货数中的√勾选符号自动去除
- "N箱"格式解析为数字N
- "N/M"格式（整箱/零散）解析为N
- 图片超10MB自动降DPI
- 无表头表格自动推断列结构
- OCR缓存机制（_ocr_cache.json），避免重复API调用

**手写标注处理**（已实现逻辑，待实际数据验证）：
- 情况1: 多送"+数字" → 加到送货数
- 情况2: 缺货"×" → 数量置0
- 情况3: 合计行后手写补充行 → 解析为新条目

### 2. Excel读取模块 (excel_reader.py)

**支持的文件类型**：
- 采购订单：表头第1行，12列
- 采购入库单：Sheet1明细(17列) + Sheet2汇总(5-6列)
- 进货单：表头第8行，遇"合计"停止
- 退货单：结构同进货单

**特殊处理**：
- 太古文件名：`PuArrival(2).xlsx太古.xlsx` = 进货单
- 翔瑞缺"核算状态"列 → 忽略
- 通力缺"应收数量"列 → 用实收数量
- `~$`临时文件自动跳过

### 3. 比对逻辑模块 (reconciler.py)

**匹配策略**（多级匹配，逐级降级）：
1. PO号 + 条码 + 仓库（精确匹配）
2. PO号 + 条码（忽略仓库）
3. 条码 + 仓库（无PO号时）
4. 仅条码（最宽松）

**一致性判定规则**：
- 数量、单价、金额全部一致 → **一致**
- 金额一致，数量/单价不同 → **一致**（标注单位换算或单价口径差异）
- 金额不一致 → **差异**（红色标注）
- 无匹配项 → **未匹配**

**退货处理**：
- 退货条目单独列出（蓝色）
- 实际应付 = 进货总额 - 退货总额
- 文件名金额用于校验

### 4. 报告生成模块 (report_generator.py)

**每个供应商生成一份Excel（5个Sheet）**：
- Sheet1 一致项（绿色底色）
- Sheet2 差异项（红/黄底色，含差异详情）
- Sheet3 未匹配项（红/灰底色）
- Sheet4 退货明细（蓝色底色）
- Sheet5 对账汇总（统计数据）

**全供应商汇总报告**：所有供应商汇总到一张表

## 已知问题与待优化

### 高优先级

1. **太古（0%一致率）**
   - 原因：可口可乐送货单是特殊的多表格嵌套格式，表头列名不标准（如"预销:02"混在表头里）
   - 建议：为太古（可口可乐）写专用解析器，或让供应商改用标准送货单格式

2. **优亿家（46.5%采购对账）**
   - 原因：部分页面（第7/12/13页）表格识别正常但条目提取为0或极少，大量数据无PO号
   - 建议：检查这几页的特殊格式，可能表格结构变化

3. **锐仕（67.6%采购，42.1%财务）**
   - 原因：327/444条OCR提取率75%，第7页提取0条；财务对账差异132条
   - 建议：检查第7页表格结构；散数列手写内容OCR识别精度低

### 中优先级

4. **炬博（58.1%采购对账）**
   - 原因：29页中提取351条（覆盖率不错），但131条差异——金额不一致的129条主要是跨PO/跨仓库的数量累加问题
   - 建议：改进无PO号页面的PO号继承逻辑（从前一页继承）

5. **所有供应商的"未匹配"项**
   - 典型原因：OCR条码识别错误（12/14位）、PDF某些页面提取失败、进货单包含多批次而送货单只有当次
   - 建议：对未匹配项增加模糊条码匹配（12位→补0尝试，14位→截取尝试）

### 低优先级

6. **手写标注验证**
   - 多送"+数字"、缺货"×"、手写补充行的逻辑已实现，但当前10家供应商的OCR数据中未检测到实际样本
   - 需要用包含手写标注的实际送货单PDF验证

7. **财务对账一致率偏低**
   - 财务对账（进货单 vs 送货单）的一致率普遍低于采购对账（入库单 vs 送货单），主要因为进货单按PO号+条码匹配时，同一编码跨多个PO的情况更多
   - 建议：考虑仅按条码匹配（不限PO号），再按金额排序贪心匹配

│   │   ├── admin_invoice_meta.py ← OCR元信息模型

### 原始系统 vs 当前系统 关键差异

| 维度 | 原始系统（images_ai） | 当前系统 |
|------|----------------------|---------|
| **OCR接口** | `RecognizeTableAccurateOCR`（相同） | 相同 |
| **表格解析** | 利用 `Cell.Type`（header/body/footer）区分区域 | 仅用 `RowTl/ColTl` 坐标重建表格 |
| **行分组算法** | **DBSCAN聚类**按Y坐标分组（>20条时）；小表格用Polygon坐标分组 | 简单按 `RowTl` 索引分组 |
| **列名识别** | **不依赖列名**，不做列名映射 | 依赖列名映射（`_match_columns`） |
| **匹配核心逻辑** | **遍历行内所有单元格**，搜索条码匹配+金额匹配 | 按固定列位置取条码、数量、单价、金额 |
| **金额匹配** | 遍历整行所有列，找到等于进货单金额的数字即匹配成功 | 从指定金额列取值比对 |
| **金额组合匹配** | 支持两个金额相加匹配（`combinations(amounts, 2)`） | 不支持 |
| **PO号提取** | `[pP9dD][0Oo]-\d+-\d+-\d+`，支持9O→PO, D0→PO等更多OCR变体 | 仅处理P0→PO |
| **数据存储** | MySQL数据库（ProcessedData表，col_0~col_23动态列） | JSON文件缓存 |
| **OCR缓存** | 按图片ID存JSON文件（`static/json/ori/{image_id}.json`） | 按供应商存JSON缓存 |
| **数字解析** | `extract_floats_from_string`：支持 `'5√'`, `'/80.40'`, `'80.40/'` | `_parse_number`：类似但覆盖面不同 |
| **未匹配处理** | 两轮匹配：第一轮PO+条码+金额；第二轮仅条码+金额（标注"PO未匹配"） | 多级降级匹配但仍依赖列位置 |

### 原始系统的核心优势

**1. 不依赖列名的匹配策略**

原始系统的 `find_matching_inventory` 函数不需要知道"第几列是条码""第几列是金额"。它的逻辑是：
```
对于进货单中的每个商品（inventory）：
  1. 按PO号找到送货单对应的OCR数据行
  2. 遍历该行的所有24列（col_0~col_23）
  3. 如果某列的值包含该商品的条码 → 条码匹配
  4. 再遍历所有列，找到值等于该商品金额的列 → 金额匹配
  5. 条码+金额都匹配 → 成功
```

这种方式天然适配所有格式的送货单，因为它不关心列的顺序和名称。

**2. DBSCAN 行聚类**

当OCR返回的单元格坐标不规则时（如手写标注、印刷偏移），原始系统用 DBSCAN 聚类算法按Y坐标自动分组行，比简单的 `RowTl` 索引更鲁棒。

**3. 金额组合匹配**

支持两个单元格的金额相加等于进货单金额的情况（如一行被OCR拆成两行）。

### 建议的改进方向

基于原始系统的分析，当前系统的主要改进方向：

1. **核心改进：采用"行内搜索"匹配策略**
   - 不再依赖 `_match_columns` 列名映射
   - OCR返回的每行数据存为 `[cell_0, cell_1, ..., cell_N]`
   - 匹配时：在行内搜索13位条码 → 在行内搜索匹配金额
   - 这将一次性解决所有供应商的列名兼容性问题

2. **增强PO号识别**
   - 扩展正则：`[pP9dD][0Oo]` 支持更多OCR变体

3. **可选：引入DBSCAN行聚类**
   - 对于复杂布局的送货单（如太古），DBSCAN可能帮助正确分组行

4. **可选：金额组合匹配**
   - 处理OCR将一行拆成两行的边缘情况

## 后续18家供应商扩展注意事项

1. **如果采用"行内搜索"策略**：新供应商零适配成本，无需扩展列名映射
2. **PO号格式**：不同供应商PO号格式可能不同，需要扩展正则
3. **单位换算**：新供应商可能有不同的箱→只换算比例，当前通过金额一致判定自动处理
4. **OCR API额度**：10家供应商约消耗~150次API调用（每页1次），18家追加约300次
5. **V3模式**：对于表格结构复杂或扫描质量一般的PDF，优先尝试V3模式

## 2026-03-26 分笔匹配Bug修复

### 问题描述

进货单同一条码分多笔入账时（如第一笔40件、第二笔8件，对应OCR送货单48件），系统在匹配第一笔后将整条OCR记录标记为已消耗，导致第二笔无法匹配，报告差异。

**典型案例**：双广供应商白象大辣椒麻辣小龙虾（条码6935270641254）
- OCR PO-0037: 48件 → 进货单拆为 40件 + 8件（不同PO号）
- 修复前：40件匹配成功，8件报"送货单未找到匹配"

### 修复方案（reconciler.py reconcile_finance函数）

1. **分笔消耗机制**：将二值`consumed_d_ids`替换为`remaining_d_qty`数量追踪
   - 每条OCR记录追踪剩余可分配数量
   - 进货单逐笔从OCR记录消耗所需数量
   - OCR数量 > 进货单数量时拆分，生成虚拟条目用于比对
   - 部分消耗后剩余数量可供后续进货单笔继续匹配

2. **两轮匹配策略**：防止条码回退抢占PO精确匹配的OCR数量
   - 第一轮：仅PO+条码精确匹配（消耗各PO对应的OCR数量）
   - 第二轮：未匹配条目走条码回退 + 行内搜索（消耗剩余数量）

3. **未匹配送货单处理**：区分完全未匹配和部分剩余
   - 完全未消耗的OCR条目仍报"进货单中未找到匹配"
   - 部分消耗后仍有剩余的报"分笔后剩余X件未匹配"
