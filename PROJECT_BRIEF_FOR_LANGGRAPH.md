# 超市采购对账自动化系统 — 项目交接文档

> 本文档用于将现有Python对账系统迁移到 LangGraph + RAG 架构时，向新项目的AI助手提供完整上下文。

---

## 一、这个系统是干什么的

校内超市（约20000名学生，3家分店：总店、中职店、高职店），每月从约28家供应商采购商品。采购助理需要核对：**供应商的送货单** 和 **用友ERP系统的入库单** 是否一致。

以前完全靠人工逐条比对Excel和纸质单据，现在这个系统把流程自动化了：
1. 把供应商的纸质送货单PDF用OCR识别成结构化数据
2. 读取用友系统导出的Excel文件
3. 两边逐条比对，输出差异报告

**当前成绩**：10家供应商，采购对账一致率88.9%，财务对账72.5%。还有18家供应商待接入。

## 二、数据流：从原始单据到对账报告

```
每个供应商有一个文件夹，里面放着：
├── 送货单.pdf            ← 纸质扫描件（供应商提供）
├── 采购订单.xlsx         ← 用友导出
├── 采购入库单.xlsx       ← 用友导出
├── 进货单.xlsx           ← 用友导出
└── 退货单.xlsx           ← 用友导出（不一定有）

处理流程：
  送货单.pdf ──→ [腾讯云OCR] ──→ 结构化条目（条码/数量/单价/金额/PO号/仓库）
                                          │
  采购入库单.xlsx ──→ [openpyxl读取] ──→ 结构化条目
                                          │
                                    ┌─────┴─────┐
                                    ▼           ▼
                              阶段一：采购对账    阶段二：财务对账
                              (送货单 vs 入库单)  (进货单 vs 送货单 vs 退货单)
                                    │           │
                                    └─────┬─────┘
                                          ▼
                                  对账报告.xlsx（带颜色标注）
```

## 三、核心业务逻辑（最重要的部分）

### 3.1 四级递进匹配

对账的本质是：拿送货单的每一条，到入库单里找对应的那一条。但数据来源不干净（OCR可能识别错、供应商格式不统一），所以匹配必须"先严后松"：

| 级别 | 匹配条件 | 为什么需要 |
|------|---------|-----------|
| 1级 | PO号 + 条码 + 仓库 | 最精确，不会错 |
| 2级 | PO号 + 条码 | 仓库名可能不统一（"总店" vs "集盒超市总店仓库"） |
| 3级 | 条码 + 仓库 | 有些供应商送货单上没有PO号 |
| 4级 | 仅条码 | PO号和仓库都没有时的兜底 |
| 兜底A | 行内搜索 | OCR把条码识别到错误的列，但那一行某个格子里确实有这个条码 |
| 兜底B | 模糊条码 | OCR把13位条码识别成12位或14位，自动补位/截取/替换一位 |

**关键规则**：每条入库单只能被匹配一次（consumed_ids机制防止重复消耗）。

### 3.2 一致性判定规则

找到匹配后，判断"一致"还是"差异"：

| 条件 | 判定 | 说明 |
|------|------|------|
| 数量+单价+金额全一致 | 一致 | 完美匹配 |
| 金额近似（差≤2%且≤1.5元） | 一致 | 太古等供应商箱价四舍五入 |
| 数量一致+单价差≤0.1 | 一致 | OCR单价识别微差 |
| 金额一致但数量/单价不同 | 一致 | 单位换算（如：箱→支，1箱=24支） |
| 条码匹配但跨PO数量不同 | 一致 | 跨采购订单差异，标注但不报错 |
| 以上都不满足 | 差异 | 需要人工核对 |

### 3.3 两阶段对账

- **阶段一（采购对账）**：送货单 vs 采购入库单 — 确认供应商送的货和仓库实际入库是否一致
- **阶段二（财务对账）**：进货单 vs 送货单 vs 退货单 — 确认财务记账和实际送货是否一致，应付=进货-退货

## 四、OCR解析 — 最复杂最脆弱的环节

### 4.1 API配置

- **服务商**：腾讯云
- **主API**：`RecognizeTableAccurateOCR`（表格识别，统一接口）
- **V3模式**：`UseNewModel=true`，用于炬博和优亿家（效果更好）
- **辅助API**：`GeneralAccurateOCR`（通用文字识别，仅用于提取表头PO号等）
- **缓存**：每个供应商文件夹下 `_ocr_cache.json` / `_ocr_cache_v3.json`，避免重复调API花钱
- **密钥**：存在 `SecretKey.csv`，优先读环境变量 `TENCENT_SECRET_ID/TENCENT_SECRET_KEY`

### 4.2 十家供应商的送货单格式差异

这是整个项目最大的复杂度来源。10家供应商，6种以上不同的送货单格式：

| 供应商 | 格式特点 | 难点 |
|--------|---------|------|
| **承运** | 标准表格，条码+品名可能合并在一列 | √勾选符号混在数字里要去除 |
| **优品味** | 标准表格，序号/条码/品名/零数/单价/金额 | PO号写成P0（零）需修正为PO |
| **优亿家** | 商品名称/条形码/数量/价格/金额 | 无PO号，需V3模式，部分页面提取失败需行内搜索补全 |
| **翔瑞** | 序号/条形码/商品名称/订单数量/销售价/金额 | 手写√标记干扰OCR |
| **锐仕** | 商品条码/件数/散数/件价/散价/金额 | 列名有换行需拆分，散价vs件价需选对列 |
| **炬博** | 商品条码/整件数/零散数/单价/金额/单支总数量 | 无表头页面需自动推断列结构，跨PO累加 |
| **鑫凯** | 商品编码/条形码/数量/单价/金额 | 多表格长页面 |
| **通力** | 类似承运格式 | 规格列中的"X"被误判为"缺货"标记 |
| **太古** | 可口可乐专用格式，多表格嵌套 | 完全定制的专用解析器 `_extract_taigu_items` |
| **汇基** | 类似承运格式 | 有退货单 |

### 4.3 已实现的OCR容错

- 条码13位校验，非标条码标记异常
- 数量为None时由金额÷单价反算
- √勾选符号自动去除
- "N箱"格式解析为数字N
- "N/M"格式（整箱/零散）解析为N
- 图片超10MB自动降DPI
- 无表头表格自动推断列结构

## 五、Excel读取 — 用友系统的4种文件

| 文件类型 | 表头位置 | 关键列 | 特殊情况 |
|---------|---------|--------|---------|
| 采购订单 | 第1行 | 单据编号、存货编码、数量、含税单价、含税金额 | — |
| 采购入库单 | 自动查找 | 采购订单号、存货编码、实收数量、含税单价、含税金额 | 翔瑞缺"核算状态"列，通力缺"应收数量"列 |
| 进货单 | 第8行 | 存货编码、数量、含税单价、含税金额、采购订单号 | 遇"合计"行停止 |
| 退货单 | 同进货单 | 同进货单 | 结构同进货单，`is_return=True` |

文件类型自动识别逻辑：先按文件名关键字匹配（"采购订单""采购入库""进货单""退货单"），失败时打开文件读内容判断。

仓库名标准化：送货单写"总店"→统一为"集盒超市总店仓库"。三家店：总店、中职店、高职店。

## 六、报告生成

每个供应商一份Excel，包含5个Sheet：
1. **一致项**（绿色）— 不用管
2. **差异项**（红/黄色）— 需要人工核对
3. **未匹配项**（红/灰色）— 送货单有但入库单没有，或反过来
4. **退货明细**（蓝色）
5. **对账汇总** — 统计数据

另外还有一份全供应商的汇总报告。

## 七、当前成绩与已知问题

### 各供应商对账成绩

| 供应商 | 采购对账 | 财务对账 | 主要问题 |
|--------|---------|---------|---------|
| 承运 | 97.3% | 97.8% | 几乎完美 |
| 优品味 | 96.0% | 92.3% | 几乎完美 |
| 通力 | 96.4% | 76.6% | 财务对账跨PO匹配问题 |
| 炬博 | 93.4% | 65.8% | 无PO号，跨PO累加逻辑复杂 |
| 鑫凯 | 92.7% | 87.2% | 良好 |
| 翔瑞 | 91.3% | 87.9% | 手写标注干扰 |
| 汇基 | 79.4% | 68.6% | 样本少，退货处理 |
| 锐仕 | 79.1% | 47.1% | 散价/件价列复杂，第7页提取失败 |
| 优亿家 | 76.0% | 54.1% | 部分页面OCR失败 |
| 太古 | 71.8% | 33.3% | 可口可乐格式太特殊 |

### 主要痛点（迁移到LangGraph/RAG时应重点解决）

1. **OCR错误无法自我修正** — 条码识别错一位就会匹配失败，目前只有简单的模糊匹配兜底
2. **每家供应商都是硬编码的特殊处理** — 接入新供应商需要写代码，没有泛化能力
3. **财务对账一致率偏低** — 跨PO匹配逻辑不够智能
4. **没有学习能力** — 同样的错误不会自动记住和修正
5. **没有人工反馈闭环** — 采购助理修正的差异项没有回流到系统

## 十、迁移到LangGraph + RAG的建议关注点

### 适合用LLM/RAG替代的部分
1. **OCR后处理** — 用LLM理解表格结构，而不是硬编码每家供应商的列名映射
2. **模糊匹配** — 用RAG存储历史匹配记录，遇到新的未匹配项时检索相似案例
3. **新供应商接入** — 让LLM自动推断送货单格式，而不是人工写解析器
4. **人工反馈学习** — 采购助理修正的差异项存入RAG，下次自动应用

### 不应该用LLM替代的部分
1. **精确匹配**（1-2级）— 条码和PO号的精确比对，用数据库查询比LLM更快更准
2. **Excel读取** — 结构固定的用友系统文件，openpyxl就够了
3. **报告生成** — 格式化输出，不需要智能

### 需要保留的业务规则
- 仓库名标准化映射（总店→集盒超市总店仓库）
- 金额近似判定（差≤2%且≤1.5元）
- 单位换算识别（金额一致但数量不同）
- 退货处理（应付=进货-退货）
- OCR缓存机制（节省API费用）
